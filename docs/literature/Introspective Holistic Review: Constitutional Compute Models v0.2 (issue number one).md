## 3. In-Depth Explanation of Core Components

**3.1 Canonical Vertex Registry (CVR)**

The CVR is the **semantic constitution** of the lattice — a fixed, 10-axis × 60-vertex registry that serves as the single source of truth for all meaning.

**Mechanisms**:
- **Immutable Structure**: Exactly 60 vertices, canonical IDs (V<N>.<M>), names, definitions, and sub-context families. No additions, removals, or renamings except via BEP amendment.
- **Sub-Context Families**: Binding extensions (e.g., calibration drift, narrative drift under V1.5). These clarify scope without creating new vertices, preserving the 60-count invariant.
- **Forbidden Synonyms/Antonyms**: Explicit prohibition of alternate phrasing, domain reinterpretations, or "creative mappings" to eliminate ambiguity.

**Applied Processes**:
- **Semantic Binding**: Every evaluator, CLI command, or simulation loads the CVR hash first. Deviation (e.g., synonym use) triggers immediate rejection.
- **Cross-Referencing**: Vertices link to constitutional anchors (e.g., V1.1 to Article 0 ψ₅/SE exclusivity).

The CVR prevents "glossary bloat" and "semantic mutation," ensuring compute operates on fixed meaning rather than evolving interpretation.

**3.2 Benchmark Implementation Verification (BIV)**

BIV is the **verification layer**, a runtime and post-run process that confirms compute fidelity to the constitution.

**Mechanisms**:
- **Proof Bundle Generation**: Every run produces: registry hash, sub-context log, reasoning trace (tied to anchors), cross-axis validation log, provenance (input/output hashes, timestamp, seed), and signature (hash chain).
- **Reproducibility Check**: Independent replay must yield identical scores using the same sub-contexts and reasoning.
- **Doctrine Compliance**: Verifies no substitutions, no omissions, no randomization, and adherence to dynamic telemetry (cross-axis deltas, ψ₅/SE logging).

**Applied Processes**:
- **Pre-Run Validation**: CLI/evaluator rejects if registry hash mismatches or sub-contexts are invalid.
- **Post-Run Audit**: BIV replays the proof bundle, flagging constitutional faults (e.g., narrative in reasoning trace) with rollback to prior state.
- **Telemetry Logging**: Records ψ₅/SE trajectories and cross-axis interactions for longitudinal drift detection.

BIV transforms scoring from subjective judgment to **verifiable constitutional execution**.

**3.4 Recovery Verification Process**

Recovery Verification is the **hysteresis gate** that prevents oscillation and false exits from emergency or rollback states.

**Step-by-Step Process**

1. **Trigger** — Emergency or rollback state entered  
2. **Monitoring Window** — Minimum 5 samples or 2 minutes (whichever longer)  
3. **Monotonic Improvement Rule** (strict)  
   - ψ₅ must be **non-increasing** throughout window  
   - SE must be **non-decreasing** throughout window  
4. **Threshold Exit** — Both must be satisfied **simultaneously** at end of window:  
   - ψ₅ ≤ 0.72  
   - SE ≥ 0.84  
5. **Failure** — Any monotonic violation → **immediate re-trigger** of emergency/rollback
